## Project Description

Graphable is an AI-first graphical service where the primary interaction is via LLM agents in chat. Users create, update, and operate graphs and dashboards with strict customer isolation via Usable workspaces. Configuration and definitions (graphs, dashboards, queries, transformers) are stored in Usable fragments. Runtime processing is event-driven using Flowcore Pathways, and TypeScript transformers run on Bun to materialize read models in PostgreSQL. Graphable can provision PostgreSQL databases in Flowcore-owned Azure accounts and bill customers via Stripe with transparent, pass-through pricing plus a 20% margin.

Graphable is a control plane. Execution runs in a separate worker service that provides serverless Bun slots for event-driven transformers and on-demand API endpoints.

## Architectural Principles

### Base-UI Component Usage

**CRITICAL**: Do NOT use the `asChild` prop with base-ui components. Base-ui components (Button, PopoverTrigger, DropdownMenuTrigger, ContextMenuTrigger, AlertDialogCancel, etc.) do not support the `asChild` pattern.

**Alternatives**:
- For Links with Button styling: Wrap the Link with button classes or use a styled Link component directly
- For triggers: Wrap the child component (e.g., Button) inside the trigger component instead of using `asChild`
- Example: Instead of `<Button asChild><Link>...</Link></Button>`, use `<Link className="...button styles...">...</Link>` or wrap the Button inside the Link

### Usable Fragment Data Storage

**CRITICAL**: All configuration data MUST be stored in the JSON content (`fragment.content`). The `fragment.title`, `fragment.summary`, and fragment frontmatter are for Usable convenience and AI search only.

- **Source of Truth**: The JSON content (`fragment.content`) is the single source of truth for all configuration data
- **Search Optimization**: `fragment.title`, `fragment.summary`, and frontmatter should be synced FROM the JSON content for better searchability
- **Zod Validation**: All configuration objects MUST have Zod schemas that validate the JSON content structure
- **Data Location**: Configuration fields like `title`, `description`, `name`, etc. should be stored in the JSON content, not just in fragment metadata

**Example**: For a dashboard:

- `fragment.content` contains `{ "title": "My Dashboard", "layout": {...}, ... }` ‚Üê **Source of truth**
- `fragment.title` should be synced to `"My Dashboard"` for search convenience
- When reading, always read from `fragment.content.title`, not `fragment.title`
